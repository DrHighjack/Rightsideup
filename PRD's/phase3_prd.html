Transaction Coordinator Role

A TC is a single login that manages orders for multiple realtors. They act on behalf of agents — invoices and ownership always belong to the agent, not the TC.

Key design decisions

A TC account is its own User record with role = TC. It is linked to one or more Realtor User records via a TCAgentLink join table. When a TC logs in, they see a context switcher — they pick which agent they're acting as, and the entire session context shifts to that agent. Orders placed by a TC are owned by the selected agent (realtorId = agent's id, not TC's id). The TC's userId is stored separately as placedByTCId for audit purposes.

TC invite flow (realtor initiates)

1
Realtor goes to /dashboard/account → "Transaction Coordinators" section → clicks "Invite TC"
2
Realtor enters TC's email address and submits
3
Server checks: does a User with that email exist with role = TC? If yes → create TCAgentLink immediately. If no → create a pending invite record and send an invitation email to that address.
4
Invite email contains a link to /register/tc?token=[invite_token] — pre-fills email, forces role = TC on registration
5
TC completes registration → TCAgentLink is created automatically → TC can now log in and see that agent
6
Realtor can revoke TC access at any time from their account page — deletes the TCAgentLink

TC login and agent switcher

1
TC logs in → redirected to /tc/select-agent — shows list of all linked agents (name, brokerage, active order count)
2
TC selects an agent → session context set to that agent (stored in session as activeAgentId)
3
TC is redirected to /tc/dashboard — identical to realtor dashboard but showing selected agent's data. Header shows "Acting as: [Agent Name]" with a "Switch Agent" button.
4
All API calls made while TC is active pass activeAgentId — server validates TC is linked to that agent before executing
5
"Switch Agent" returns to /tc/select-agent without logging out

New API Routes

TC management

POST/api/tc/inviteRealtor sends TC invite by email. Creates TCInvite record, sends email. Auth: REALTOR.
GET/api/tc/invite/[token]Validates invite token. Returns: valid/expired, inviting realtor name. Used on /register/tc page.
GET/api/tc/agentsTC gets list of their linked agents. Auth: TC only.
POST/api/tc/set-agentTC sets activeAgentId in their session. Body: { agentUserId }. Validates link exists.
GET/api/dashboard/tc-linksRealtor sees their linked TCs. Auth: REALTOR.
DEL/api/tc/links/[id]Realtor or Admin removes a TC link. Realtor can only remove their own links.

Admin TC management

GET/api/admin/tcsAll TC accounts with their linked agent count.
POST/api/admin/tcs/linkAdmin manually links a TC to an agent. Body: { tcUserId, agentUserId }.
DEL/api/admin/tcs/link/[id]Admin removes any TC-agent link.

Sign inventory — admin

GET/api/admin/signsAll signs. Filters: status, type, assignedToUserId. Includes summary counts by status.
GET/api/admin/signs/[id]Single sign detail with assignment history.
POST/api/admin/signsCreate sign. Body: signNumber, type, notes, purchasedAt.
PUT/api/admin/signs/[id]Update sign — status, assignment, location, notes. Assigning to an order sets deployedAddress from order address.
POST/api/admin/signs/bulkBulk create signs. Body: { type, count, prefix } — generates sequential signNumbers.
GET/api/admin/signs/mapReturns all DEPLOYED signs with lat/lng for map view.

Sign inventory — realtor / TC

GET/api/signs/mineSigns assigned to the logged-in realtor (or active agent if TC). Status, location, order.
POST/api/signs/[id]/reportRealtor reports a sign lost or damaged. Body: type, description. Creates SignReport, emails admin.
POST/api/signs/reorderRealtor requests more signs. Body: type, quantity, notes. Sends email to admin — no auto-fulfillment.

Low Inventory Alert

Server-side check triggered on sign status changes.

After any sign status update that reduces AVAILABLE count, run a check: if AVAILABLE signs of any type drops below the threshold (default: 5 per type), send an admin alert email. Use the existing Nodemailer setup from Phase 1. Store the threshold as an env var LOW_INVENTORY_THRESHOLD defaulting to 5. Do not send duplicate alerts — track lastAlertSentAt per sign type, don't re-alert within 24 hours.

  • Check runs after: sign status → DEPLOYED, DAMAGED, LOST, RETIRED
  • Email to: admin email (from env var ADMIN_ALERT_EMAIL)
  • Email body: sign type, current available count, threshold
  • Admin dashboard also shows a warning badge on the Signs nav item when any type is low

Acceptance Criteria

TC inviteRealtor enters a new email, invite is sent, TC registers via the link, and TCAgentLink is created. Realtor's account page shows the TC as linked.
TC existing userRealtor enters email of an existing TC account — TCAgentLink is created immediately with no invite email sent.
TC login flowTC logs in, sees their linked agents, selects one, lands on TC dashboard showing that agent's orders and data with the "Acting as" banner visible.
TC order placementTC places an order while acting as Agent A. The order's realtorId = Agent A's id. placedByTCId = TC's id. Order appears in Agent A's dashboard.
TC invoice accessTC can view Agent A's invoices. The pay button is not visible to the TC.
TC revokeRealtor revokes TC access. TC can no longer see that agent in their agent list. Existing orders are unaffected.
Sign inventoryAdmin can add a sign, assign it to an order, and the sign appears in the realtor's /dashboard/signs page with the correct address and status.
Bulk addAdmin bulk-adds 10 "Standard" signs with prefix "SPF-S". Signs SPF-S-001 through SPF-S-010 are created with status AVAILABLE.
Map viewAdmin /admin/signs/map shows a pin for every DEPLOYED sign. Clicking a pin shows sign number, realtor, and address.
Sign reportRealtor reports a sign as lost. Admin receives an alert email. Sign status changes to LOST. Report appears in /admin/signs under Reports tab.
Low inventory alertWhen available "Standard" signs drop below threshold, admin receives one alert email. A second status change within 24 hours does not send a duplicate.
TC isolationTC acting as Agent A cannot see Agent B's data even if also linked to Agent B — they must switch agents first via /tc/select-agent.
