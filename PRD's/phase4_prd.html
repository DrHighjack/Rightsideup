02

811 Email Monitoring

A background service polls a dedicated 811 inbox, parses incoming notifications, and triggers order holds automatically.

How 811 emails work

When a sign install order is placed near a utility dig site, the 811 notification center emails a designated address with the ticket number, excavation address, and work start date. The parser needs to extract the street address from this email and match it against pending/scheduled orders within a configurable radius (default: 0.1 miles). Emails come from various senders depending on state — always check the subject line for "811" or "Dig Safe" or "One Call" keywords.

Email polling setup — two options

Option A — IMAP (recommended)
  • Use imapflow npm package
  • Works with any email provider (Gmail, Outlook, custom)
  • Poll every 5 minutes via node-cron
  • Mark emails as read after processing
  • No OAuth required — app password works
  • Simpler setup, more reliable
Option B — Gmail API
  • Use Gmail API with service account
  • More complex OAuth setup
  • Better for Gmail-specific features
  • Push notifications instead of polling
  • Use only if IMAP is blocked by provider

Build with Option A (IMAP + imapflow) by default. The admin can configure the inbox credentials in settings. If they're using Gmail, they'll need to enable IMAP and generate an App Password in Google Account settings — not their regular Gmail password.

811 workflow

1
Scheduler fires — node-cron runs every 5 minutes, calls the 811 poller service
2
Fetch unread emails — IMAP connects to configured inbox, fetches unread messages, filters by subject keywords: "811", "Dig Safe", "One Call", "utility locate", "ticket"
3
Parse address — Extract street address from email body using regex patterns. If confident match found → proceed. If uncertain → create ticket with status NEEDS_REVIEW and alert admin.
4
Cross-reference orders — Query orders with status PENDING or SCHEDULED where address contains the parsed street name and city. Flag matches.
5
Create Ticket811 record — Save ticket with: raw email, parsed address, ticket number, matched order IDs, status
6
Hold matched orders — Update matched order status to ON_HOLD, set holdReason = "811 ticket [number]"
7
Alert admin — Send email to ADMIN_ALERT_EMAIL with ticket number, address, matched orders, and link to /admin/811/[id]
8
Mark email read — Mark processed email as read in IMAP so it isn't processed again

Order status additions

Add ON_HOLD to the Order status enum. This is a new status that sits between SCHEDULED and IN_PROGRESS. Orders on hold are not assigned to field techs and do not appear in the job queue. Add holdReason (String?) and heldAt (DateTime?) fields to the Order model.

Order status flow with 811

PENDING → SCHEDULED → ON_HOLD → SCHEDULED → IN_PROGRESS → COMPLETED

Admin 811 release flow

1
Admin receives clearance from utility company (separate process, outside the app)
2
Admin goes to /admin/811/[id] → clicks "Mark Cleared"
3
Ticket811 status → CLEARED. All linked orders with holdReason matching this ticket → status returns to SCHEDULED, holdReason cleared
4
Admin and realtor receive email confirming orders are back in queue

Background Scheduler

Runs recurring tasks server-side using node-cron.

In Next.js App Router, initialize the scheduler in a custom server file or use a Route Handler that's called on startup. The cleanest approach for this stack is to create /lib/scheduler.ts and initialize it in the Next.js instrumentation hook (instrumentation.ts in the root) — this runs once on server startup in both dev and production.

Acceptance Criteria

811 pollingScheduler runs every 5 minutes. Admin can trigger manual poll from /admin/811. New emails in the 811 inbox create Ticket811 records within one poll cycle.
Address matchAn order with address "123 Main St, Seattle WA" is automatically put ON_HOLD when a 811 ticket arrives for "123 Main Street, Seattle". Street name matching is fuzzy — not exact string match.
No matchA 811 ticket with an address that doesn't match any active orders creates a ticket with status ACTIVE and zero matched orders. No orders are put on hold.
Parse failureAn email where address cannot be confidently extracted creates a ticket with status NEEDS_REVIEW. Admin receives alert email. No orders are put on hold.
Admin alertAdmin receives email within one poll cycle of a new 811 ticket being created with matched orders.
Clear ticketAdmin marks ticket CLEARED. All orders with holdReason referencing that ticket number return to SCHEDULED status. Realtor receives email that their order is back in queue.
Field tech loginFIELD_TECH user logs in and is redirected to /field/dashboard. They cannot access /admin/* or /dashboard/* routes.
Job assignmentAdmin assigns a SCHEDULED order to a field tech via /admin/jobs. Job appears in field tech's dashboard immediately.
Job completionField tech marks job complete from /field/jobs/[id]. Order status changes to COMPLETED. If QB is connected, invoice is auto-created.
Issue flagField tech flags an issue on a job. Admin receives email with job details and issue description. Job is not marked complete.
Settings saveAdmin saves IMAP credentials in /admin/settings. "Test Connection" button confirms the connection works. Scheduler uses the saved credentials on next poll cycle.
Invoice agingInvoices overdue by 7 days receive a reminder email to the realtor. Same invoice does not receive a second 7-day reminder — only escalates to 14-day reminder next.
